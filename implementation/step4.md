# Step 4: Choosing a data format [DL]
Opening many-column-datasets like PSID (with long-run merged waves and supplements ~2,000–10,000+ variables!), Compustat (expanded + ratios + segments + historical fields: 1,000–3,000 variables!), or SILC-EU (~300–1,500 variables depending on module) in formats such as CSV or Stata can be very slow on an 8 GB machine because these formats typically load a large portion of the dataset into memory, even if you only need a small subset of variables. Since these datasets are wide (many variables across individuals and waves), loading them means the system has to read, parse, and store far more data than is actually used in the analysis. This quickly consumes RAM, and once memory is exhausted, the computer starts turning to hard disk, which is much slower and causes freezing. The problem is that these formats do not efficiently allow you to skip unused columns during reading, so performance is driven by the full dataset size rather than the subset you actually need for econometric work.

With these issues in mind, I came across Paul Goldsmith-Pinkham’s mini series on Claude Code for applied economists, and discovered Parquet format. What stood out immediately was the format’s ability to embed rich metadata directly within the dataset itself. This would allow me to inspect variable definitions or a codebook on the fly during preprocessing or analysis, without needing to switch to an external documentation file (I will expand on this in the next step).

Second thing that impressed me was the size of the compression: If a dataset is 10 GB as CSV, it will often become ~1–3 GB in Parquet! So on an 8 or 16 or even 32 GB RAM machine, this matters a lot. CSV/DTA often does not fit comfortably in memory once loaded, while Parquet often fits easily or can be partially loaded calling only needed columns without reading the entire file. So go Parquet!
