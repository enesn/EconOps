Every time I build the dataset, I also produce a written record of where that particular version came from and how it was made. I treat this record as part of the output itself so that anyone (including me, later) can trace any figure in the final file back to its origins and, if needed, rebuild it exactly.
The metadata record, a YAML file, has a few layers. First, I identify the raw source material precisely. Rather than just naming the input files, I compute a unique fingerprint (SHA hashes) for each one. This value  changes if even a single character of the file changes. I store those fingerprints alongside the file names and descriptions. This means I can always confirm that a given build was made from exactly the inputs I think it was, and detect if a source file has since been altered or replaced.

Second, I capture the processing steps as they actually ran. I record which version of the code produced the build, which scripts executed and in what order, and the software environment they ran in — the language version, the specific versions of every package I relied on, and the computing image. Because software and dependencies change over time, pinning all of this is what makes a build genuinely reproducible.

Third, I describe the output itself: its structure, the number of records and variables, what each record represents, how it is uniquely identified, and which variables serve special roles. I produce and keep a companion codebook that documents every variable, what its coded values mean, and — importantly — which original source variable was used to construct it in each survey wave. Since the underlying survey renames and re-codes its variables over time, this cross-wave mapping is essential for anyone who wants to understand how a harmonised variable was assembled across decades of data.

Finally, I keep an honest account of quality and change. I note known limitations and discrepancies against the reference version I validate against, and I maintain a running log of what changed between releases and why — for example, when I extend the file to a new survey wave, I record which coding rules had to change and the specific reasons behind each fix.
The guiding principle is that the data and its history should travel together.
